# CharacterCadre — Project Development Guide

> **Audience:** Claude Code (executor) and the human reviewer (auditor).
> **Goal:** Build CharacterCadre — a browser-based local app that uses Ollama to run a structured, multi-phase RPG roleplay loop with a Dungeon Master and characters.

---

## Tech Stack (Decided)

| Layer | Choice | Rationale |
|---|---|---|
| Backend | **Python 3.11+ / FastAPI** | First-class Ollama client, clean SSE streaming, Pydantic validation, structured outputs map directly to JSON schema. |
| LLM Runtime | **Ollama** (native API, not OpenAI-compat) | Better structured-output support via native `format` param. |
| Frontend | **React + Vite + TypeScript** | Streaming UX, type safety for editor forms. |
| Styling | **Tailwind CSS** | Fast iteration, no design overhead. |
| Storage | **JSON files on disk** | Per user request. SillyTavern-compatible character cards (PNG/JSON). |
| Process model | Single-user localhost, no auth | Per user request. |

## Core Domain Models (used throughout — define once in Stage 1)

```python

{
  "id": "uuid",
  "name": "string",
  "description": "string",        # personality, may use {{user}}/{{char}}
  "description_summary": "string",  # one short sentence, auto-generated by LLM, user-editable. Used in Phase 1 director.
  "description_hash": "string",   # hash of `description` at time summary was generated; used to detect when summary needs regeneration
  "response_examples": [          # list of {user_msg, char_msg} pairs
    {"user": "string", "char": "string"}
  ],
  "is_dm": false,
  "avatar_path": "string|null"    # optional
}

# Scenario
{
  "id": "uuid",
  "name": "string",
  "summary": "string",            # short paragraph for Director orientation. Auto-generated, user-editable. Optional — falls back to initial_message excerpt.
  "summary_hash": "string",       # hash of (initial_message + system_prompt) at time summary was generated; used to detect when regen is needed
  "initial_message": "string",    # DEPRECATED-ish: still supported for beatless scenarios. If beats are defined, beat[0].starter_prompt is used instead.
  "system_prompt": "string",      # default RP prompt prefilled
  "persistent_messages": ["string"],  # always shown to LLM
  "dm_only_info": ["string"],         # ONLY shown to DM character
  "recommended_character_ids": ["uuid"],
  "beats": [                          # OPTIONAL. Empty list = beatless scenario (current behavior).
    {
      "id": "uuid",
      "order": 0,                     # zero-indexed; beat[0] is the opener
      "name": "string",               # e.g., "Town", "Dungeon Entrance"
      "description": "string",        # DM/author notes — environment, hazards, minor NPCs. Shown to Director and (when active) to DM.
      "summary": "string",            # one-sentence summary used in the Director's beat roster, so it can decide whether to skip ahead. Auto-generated by summarizer module, user-editable.
      "summary_hash": "string",       # hash of (name + description + transition_condition) for regen detection
      "transition_condition": "string",  # plain English: "the player physically leaves the town gates"
      "starter_prompt": "string"      # prose injected as a DM message when this beat begins
    }
  ]
}

# Save (game state)
{
  "id": "uuid",
  "scenario_id": "uuid",
  "name": "string",                # auto-generated or user-named
  "active_character_ids": ["uuid"],
  "user_name": "string",           # for {{user}} substitution
  "current_beat_id": "uuid|null",  # null if scenario has no beats, or if sandbox_mode is true
  "sandbox_mode": false,           # set true after final beat completes; disables beat detection
  "messages": [
    {
      "id": "uuid",
      "role": "user|character|dm",
      "character_id": "uuid|null", # null for user
      "content": "string",
      "timestamp": "iso8601",
      "is_dm_only": false,         # rare, but supported
      "beat_id_at_time": "uuid|null"  # which beat was active when this message was created (for debug + UI)

  "updated_at": "iso8601"
```

## Phase Pipeline (the heart of the app)

For every user turn:

1. **Phase 0** — User submits message. Backend appends to save's `messages`.
2. **Phase 1 — Director:** Run a meta-level, non-streaming structured call using the **Director Prompt Assembly**. The Director sees a neutral systemic prompt, the scenario summary, the beat roster (current beat + future beats with summaries), a roster of characters with their summaries, and the chat history. It returns JSON with the chosen speaker, an optional beat transition signal, and a per-turn direction note for the DM. The Director is NOT in-character — it is a routing/strategic decision.
   - **Beat transition handling (server-side, between Phase 1 and Phase 2):** If the Director signaled `beat_transition=true`, the server advances `save.current_beat_id` to the chosen `next_beat_id` and appends the new beat's `starter_prompt` as a DM message in the chat history. The DM's upcoming response in Phase 2 will see this as the most recent context.
3. **Phase 2 — In-narrative response:** If `dm_should_narrate=true`, stream a DM response first (DM Prompt Assembly, full context, includes the Director's direction note). Then stream the chosen character's response (Character Prompt Assembly, restricted context).
4. **Phase 3 — Player option drafting:** Run a structured call using the DM Prompt Assembly + an instruction to produce 4 plausible player responses. Frontend renders as discrete clickable choices (plus a free-text input as escape hatch).
5. Loop back to Phase 0.

### Director Prompt Assembly (Phase 1 — speaker routing)

The Director is a **meta-level, systemic** decision-maker, not an in-fiction voice. Its prompt is deliberately neutral and excludes anything that would inject narrative personality.

```
[Director system prompt — hardcoded constant, neutral]
   "You are the Director of a roleplaying game. Your only job is to decide which
    character should speak next based on what just happened. You do not narrate,
    write dialogue, or take a character's voice. You return a structured decision."

[Scenario.name]
[Scenario.summary]   ← short one-paragraph orientation; falls back to first ~200 chars of initial_message if absent

[Active character roster — formatted as bulleted lines]
   - <name> (DM): <description_summary>
   - <name> (companion): <description_summary>
   - ...

[Beat roster — only included if scenario has beats AND save is not in sandbox_mode]
   "The story is structured in the following beats:
    - [CURRENT] Beat 2: Dungeon Entrance — <summary>. Transition: <transition_condition>
    - Beat 3: Boss Chamber — <summary>. Transition: <transition_condition>
    - Beat 4: Escape — <summary>. Transition: <transition_condition>
    You may signal a transition to the next beat OR skip ahead to a later beat if the
    story has clearly moved past intervening events. You cannot go backward."

[Scenario.persistent_messages — joined]
[Scenario.dm_only_info — joined]   ← Director has full visibility, like DM

[Truncated shared chat — same DM-level visibility, includes DM-only messages]

[Director instruction — hardcoded constant]
   "Decide:
    1) Who speaks next (set speaker_character_id; set dm_should_narrate=true if the
       scene needs narration before a character responds).
    2) Whether the current beat's transition condition has been met. If yes, set
       beat_transition=true and choose next_beat_id (the next beat by default, or a
       later one if the story has skipped ahead).
    3) Write a brief direction_note (1-2 sentences) for the DM that summarizes what
       this turn should accomplish narratively. This is internal guidance, not dialogue.
    Return JSON only."
```

**Excluded from the Director prompt** (these are for in-character work, not routing):
- `Scenario.system_prompt` (that's the in-character RP prompt)
- The DM character's `description` and `response_examples`
- Any non-DM character's `description` or `response_examples`
- Beat `description` and `starter_prompt` fields (Director only sees beat summaries; the DM gets the description when its beat is active)

The director only needs `description_summary` per character and `summary`/`transition_condition` per beat to do its job. Including full descriptions would bloat the prompt, bias the director toward verbose entries, and risk the director picking up narrative voice.

### DM Prompt Assembly (Phase 2 when DM narrates, Phase 3 player options)

```
[Scenario.system_prompt]
[DM Character.description]
[DM Character.response_examples — formatted as {{user}}/{{char}} pairs]
[Scenario.persistent_messages — joined]
[Scenario.dm_only_info — joined]   ← DM ONLY

[Current beat context — only included if save has an active beat]
   "Current scene: <beat.name>
    <beat.description>"

[Truncated shared chat — last N turns that fit budget]

[Director's note for this turn — only included if Director provided one, appended after chat]
   "Director's note for this turn: <direction_note>"
```

The Director's note appears as a final instruction, after the truncated chat, so it's the most recent and salient context the DM sees before generating its response. It guides without dictating — the DM still writes the actual prose.

The beat's `starter_prompt` is **not** in this assembly. When a beat begins, the starter prompt is appended to the chat as a real DM message; from then on it's just normal chat history.

### Character Prompt Assembly (Phase 2 when a non-DM character speaks)

```
[Scenario.system_prompt]            ← shared system prompt is fine to share
[Character.description]
[Character.response_examples]
[Scenario.persistent_messages]
[Truncated shared chat — DM-only messages stripped]
```

**Excluded from non-DM character context:** DM-only info, DM-only chat messages, other characters' description fields, scenario setup text marked DM-only.

### Token budget rule

`available_chat_tokens = max_context_tokens - tokens(everything_above_chat) - response_reserve(default 1024)`

Truncate from the **top** (oldest first), preserving the most recent turns. Always keep the most recent user message.

> Variable substitution rules (`{{user}}`, `{{char}}`) are documented in the Resilience & Logging section below, since they vary by prompt type.

## Story Beats (scenario-driven plot progression)

Beats are an **optional** structuring mechanism for scenarios. A scenario with no beats behaves exactly like the pre-beats system. A scenario with beats gets pre-scripted plot progression, controlled by the Director.

### Lifecycle

1. **Authoring**: In the scenario editor, an author lays out beats as an ordered list. Each beat has a `name`, `description` (DM notes about the environment), `summary` (one-sentence Director hint, auto-generated like character summaries), `transition_condition` (plain English check), and `starter_prompt` (prose that opens the beat).

2. **Game start**: A new save with a beats-enabled scenario sets `current_beat_id = beats[0].id`. The first beat's `starter_prompt` is appended as the opening DM message. (This replaces `Scenario.initial_message` for beats-enabled scenarios.)

3. **Per turn (during gameplay)**: The Director receives the beat roster — every beat from `current_beat_id` onward, with the current one marked `[CURRENT]`. Each entry shows name, summary, and transition condition. The Director judges whether the current beat's transition condition has been met based on chat history.

4. **Transition signaled**: Director returns `beat_transition=true` and `next_beat_id`. The server:
   - Validates `next_beat_id` is a forward beat (current or any future beat — no backward jumps).
   - Updates `save.current_beat_id`.
   - Appends the new beat's `starter_prompt` as a DM message in the chat (with `beat_id_at_time` set to the new beat).
   - Continues to Phase 2, where the DM will see the starter prompt as the most recent context and respond accordingly.

5. **Beat skipping**: The Director can choose any forward beat as `next_beat_id`, not just the immediate next. If the player clearly leapt ahead in the story, the Director may skip beat 2 and jump straight to beat 3. The skipped beat's `starter_prompt` is not injected — skipping means skipping.

6. **Final beat completion**: When the Director signals a transition out of the last beat, the server sets `sandbox_mode=true` and `current_beat_id=null`, then surfaces an "ending screen" event to the frontend. The player can choose to end the game or continue in sandbox mode (Director stops checking for transitions; gameplay continues without beat structure).

7. **Beatless scenarios**: If `Scenario.beats` is empty, the Director never sees a beat roster, never signals transitions, and `current_beat_id` stays `null`. Everything works as before.

### Direction Note

On every Director call (with or without beats), the Director writes a 1-2 sentence `direction_note` for the DM. This is the Director's strategic guidance — what should this turn accomplish? — passed into the DM's prompt as a final instruction. Examples:

- "The player just opened the trapped door. The DM should describe the trap triggering and Bram's reaction, leaving room for the player to attempt to dodge."
- "Tension has been building for three turns without resolution. The DM should escalate by introducing the dragon's distant roar."
- "Beat just transitioned to the Boss Chamber. The DM should establish the room's scale and the boss's presence."

> If we observe in playtesting that per-turn direction notes harm DM coherence (the DM may follow the note too literally and lose its own narrative flow), the fallback is to only inject `direction_note` on beat transitions. This is a tunable behavior and worth flagging in audit.

### Beat Control (player-facing menu)

The chat view includes a "Beat Control" option (gear icon → submenu) with:

- **Current beat indicator**: shows the active beat name as a chat-view header (e.g., "Act II: The Dungeon"). Players see the name only; transition conditions and starter prompts stay hidden.
- **Manual beat advance**: a list of forward beats the player can jump to. Two modes:
  - *Soft advance*: server appends the chosen beat's `starter_prompt` as a DM message, sets `current_beat_id`, and gameplay continues with full chat history intact. Same as a Director-driven transition.
  - *Hard advance with context wipe*: server truncates `save.messages` to just the new beat's `starter_prompt` as the opening DM message. Save metadata, character roster, and `user_name` are preserved. Useful for recovery if the LLM has gone off the rails or the player wants to skip past a stalled section.
- **Sandbox mode toggle**: lets the player exit beat tracking entirely.

These controls bypass the Director and update state directly. They exist for recovery and player agency, not as the primary progression mechanism.

### Edge cases and rules

- **Stuck sessions**: If the Director never signals a transition and the player feels stuck, manual beat advance is the escape hatch. Per spec, we are not building automatic stuck-detection in MVP.
- **Invalid `next_beat_id`**: If the Director returns a beat ID that doesn't exist or is earlier than the current beat, the server logs a warning, ignores the transition signal, and proceeds with the rest of the Director's response (speaker, direction note).
- **Editing beats mid-game**: If the author edits a scenario's beats while a save based on it is in progress, the save uses the latest beats by ID. If `save.current_beat_id` no longer exists in the scenario (the author deleted it), fall back to `null` and warn at game load. If the beat content changed but the ID remains, the save just picks up the new content on its next read — no migration needed.
- **Beat hash for summary regen**: A beat's `summary_hash` is over `name + description + transition_condition`. Edit any of these and the summary regenerates in the background, just like character summaries.

## Resilience & Logging (cross-cutting infrastructure)

Local LLM output is unreliable. Models hallucinate, return malformed JSON, repeat themselves, hang, and produce empty responses. CharacterCadre treats every Ollama call as untrusted I/O that must be validated, retried, and logged.

### Validation & auto-regeneration

For every structured Ollama call (Director Phase 1, Phase 3 option drafting, summarizer), the flow is:

1. Send the call.
2. Receive the raw response.
3. Pass it through a validator. Validators are Pydantic models that match the expected schema *plus* additional structural checks (e.g., "options array has exactly 4 items", "speaker_character_id is in the active roster").
4. **On validation failure**: log a warning with the failure reason and the raw response, then automatically regenerate. Up to 2 retry attempts (3 total tries). Each retry appends an instruction reminder to the prompt: `"The previous response failed validation: <reason>. Return ONLY valid JSON matching the schema."`
5. **After exhausting retries**: fall back to a hardcoded safe default and emit a `validation_failed` event. The Director fallback is "DM narrates, no beat transition, empty direction note." The option-drafting fallback is 4 generic neutral options ("Wait and observe.", "Ask a question.", "Take action.", "Step back."). Summarizer fallback is empty string (Phase 1 already handles empty summaries gracefully).

For every streaming Ollama call (DM narration, character speech), validation is lighter:

1. Stream tokens as they arrive.
2. Buffer the full response.
3. Run **post-stream sanity checks**: not empty, not a near-duplicate of the immediately previous assistant message in this save (loop detection — see below), not just whitespace.
4. **On failure**: log a warning, discard the buffered response, automatically regenerate (up to 2 retries). The frontend sees this as a stream restart — emit a `regenerate` SSE event so the UI can show "Regenerating..." and clear the partial output.
5. **After exhausting retries**: emit the buffered output anyway with a `validation_warning` SSE event. Better to show the user a flawed response than a silent failure.

Validation logic lives in `validation.py`. Each validator is a small pure function that takes the raw response and returns either `Ok(parsed)` or `Err(reason)`.

### Loop detection

After every streaming response, compare the new content against the previous assistant message in the save (any role: DM or character). If similarity exceeds a threshold, treat it as a loop and trigger regeneration.

Algorithm (kept simple for MVP): normalize both texts (lowercase, strip whitespace and punctuation), then check:

- **Exact match**: triggers immediate regen.
- **>85% character-level similarity** via `difflib.SequenceMatcher.ratio()`: triggers regen.
- **>70% similarity AND same speaker as previous turn**: triggers regen (a character repeating itself with minor variation).

After 2 regen attempts, give up and emit the response with a warning. Loop detection only compares against the immediately previous message — it does not scan further back.

### Timeouts

Every Ollama call has a configurable timeout (env var `OLLAMA_TIMEOUT_SECONDS`, default 60s for streaming, 30s for structured calls).

- **Mid-stream timeout**: if no token arrives for `OLLAMA_STREAM_IDLE_SECONDS` (default 20s), abort the stream, log a warning, and trigger regeneration via the same retry logic above.
- **Total timeout**: hard cap on call duration. Aborts the call entirely, returns the safe fallback after retry exhaustion.

### Logging requirements

Logs go to stdout in plain text, prefixed with timestamp, level, and a short module/component tag. No structured logging library required for MVP — Python's `logging` module with a basic formatter is enough. The configuration lives in `logging_config.py` and is initialized at app startup.

The level is controlled by env var `LOG_LEVEL` (default `INFO`).

**INFO-level events** (always visible in production):
- App startup: model in use, Ollama base URL, data directory paths, port.
- Save lifecycle: created, loaded, deleted (with save ID and scenario name).
- Game turns: `turn started (save=<id>, beat=<name|none>)`, `turn completed (save=<id>, duration=<ms>)`.
- Beat transitions: `beat transition: <old_name> → <new_name> (save=<id>, trigger=director|manual_soft|manual_hard)`.
- Validation failures and retries: `validation failed (call=director, reason=<...>, attempt=<n>/3)`.
- Loop detection triggers: `loop detected (save=<id>, similarity=<float>), regenerating`.
- Fallback engagement: `fallback engaged (call=<...>, reason=<...>)`.
- Background summary jobs: `summary regenerated (kind=character|scenario|beat, id=<...>, duration=<ms>)`.
- Errors during HTTP request handling.

**DEBUG-level events** (visible with `LOG_LEVEL=DEBUG`):
- Full assembled prompts (Director, DM, Character, Phase 3) before sending to Ollama.
- Token budget calculations: `tokens used: system=<n>, chat=<n>, available=<n>, truncated=<m> messages`.
- Raw Ollama responses for structured calls (truncated at 500 chars).
- SSE events emitted, with payload sizes.
- Background task scheduling: `scheduled summary regen (kind=<...>, id=<...>)`.

**WARNING-level events**:
- Ollama unreachable.
- Validation final-fallback engaged (after retries exhausted).
- Beat ID validation rejection.
- Summary regen failure.
- Mid-stream timeout.
- Save file referenced a deleted character or scenario.

**ERROR-level events**:
- Unrecoverable storage write failures (atomic write rename failed, etc.).
- Unhandled exceptions in route handlers.

Every log line should be self-sufficient — no log message should require previous lines to interpret. Include the save ID, character ID, or scenario ID as appropriate so logs can be filtered by entity.

### `{{user}}` and `{{char}}` substitution scope

A clarification, since the addition of multiple prompt types raised this question:

- **Character & DM Prompt Assemblies**: substitute `{{user}}` → save's `user_name`, `{{char}}` → the active speaker's name.
- **Director Prompt Assembly**: substitute `{{user}}` → save's `user_name`. `{{char}}` is undefined here (no single speaker) — leave the literal string in place if it appears, and log a DEBUG warning. Author-written content shouldn't put `{{char}}` in scenario summaries or beat descriptions, but if they do, it's their problem to fix.
- **Phase 3 option drafting**: substitute both `{{user}}` and `{{char}}` (the latter to whichever character spoke last in the chat).

# Stage 1 — Vertical Slice: Hardcoded Chat with DM + Companion

**Goal:** End-to-end prompt → Ollama → streaming response works, with a hardcoded scenario and two hardcoded characters. **No phase logic yet.** Just: user types → backend builds a prompt for one character → response streams back.

### Tasks

1. **Project scaffolding**
   - Initialize backend with `pyproject.toml` (FastAPI, uvicorn, ollama, pydantic, pytest).
   - Initialize frontend with Vite (React + TS template), add Tailwind.
   - Add `README.md` with run instructions.
   - Add `.gitignore` covering `data/`, `node_modules/`, `__pycache__/`, `.venv/`.

2. **Hardcoded fixtures** (Python module `fixtures.py`)
   - One scenario: a basic D&D dungeon-crawl. ~3 sentence initial message, default RP system prompt, 1–2 persistent messages, 1 DM-only info string ("The dungeon contains a fire trap on the third room"), and a hand-written `summary` (~1-2 sentences orienting the Director, e.g., "A small party explores the abandoned Ironroot Dungeon, a dwarven ruin rumored to hide a relic. The dungeon is dangerous but not lethal — focus on tension and discovery.").
   - Two characters: a DM ("The Narrator") and a companion NPC ("Bram, a gruff dwarven fighter"). Both with description + 2 response examples each.
   - Both characters include a hand-written `description_summary` and a matching `description_hash`. (The summarizer module that auto-generates these is built in Stage 3; Stage 2 needs the field populated to exercise the Director prompt cleanly.)

3. **Logging setup** (`logging_config.py`)
   - Initialize Python's `logging` module with a stdout handler and a formatter: `<timestamp> <LEVEL> [<module>] <message>`.
   - Read `LOG_LEVEL` from env, default `INFO`.
   - Wire it up in `main.py` at startup. Log a startup banner: model name, Ollama URL, port, data directory.
   - All subsequent modules use `logger = logging.getLogger(__name__)`.

4. **Ollama client** (`ollama_client.py`)
   - Wrapper using `ollama.AsyncClient`.
   - Method: `stream_chat(model, messages, options)` → async generator yielding token strings, with idle-timeout enforcement (default 20s between tokens, env-configurable).
   - Method: `structured_chat(model, messages, schema, options)` → returns parsed dict, with total-timeout enforcement (default 30s, env-configurable). For Stage 1 this method is stubbed but the timeout/error-handling skeleton is real.
   - On timeout or connection failure, raises a typed exception (`OllamaTimeoutError`, `OllamaUnreachableError`) that downstream code can catch.
   - Configurable model name via env var `OLLAMA_MODEL` (default `llama3.1:8b` — adjust to whatever the user has). Configurable base URL via `OLLAMA_BASE_URL` (default `http://localhost:11434`).

5. **Prompt builder** (`prompt_builder.py`)
   - Function `build_character_prompt(character, scenario, save, user_name) -> list[ChatMessage]` returning Ollama-formatted messages.
   - Function `apply_variables(text, user_name, char_name) -> str` (with `char_name=None` allowed for Director prompts; in that case `{{char}}` is left in place and a DEBUG warning is logged).
   - Apply token budget truncation (use `tiktoken` for approximate counting, or Ollama's tokenize endpoint if available — pick one and document the choice in the code).
   - Log assembled prompts at DEBUG.

6. **Backend chat route** (`/api/chat/stream`, POST, SSE)
   - Input: `{user_message, character_id}` — for Stage 1, `character_id` is just "bram" (hardcoded).
   - Builds prompt for Bram only (DM not invoked yet).
   - Streams response tokens as SSE events: `event: token\ndata: <token>\n\n`. Final event: `event: done\n`.
   - Wraps the Ollama stream call in basic error handling: on `OllamaUnreachableError` emit `event: error\ndata: {"reason": "ollama_unreachable"}` and close the stream.
   - Note: this endpoint is superseded by `/api/chat/turn` in Stage 2. It remains in the codebase for Stage 1's smoke test only and can be deleted (or left dormant) once Stage 2 ships.

7. **Save persistence (minimal)**
   - In-memory or single JSON file `data/saves/stage1.json`.
   - Append user message and assistant response after stream completes.
   - Even at this stage, define the full Pydantic models for Character, Scenario, Beat, Save, and Message — they will be reused throughout. Don't ship a stripped-down model and refactor later.

8. **Frontend chat UI**
   - Single page: chat log + input box + send button.
   - On send: POST to `/api/chat/stream`, render streamed tokens live.
   - Display sender name above each message.
   - Loads chat history from `/api/saves/stage1` on mount.
   - Surface `event: error` with a simple toast.

9. **Tests** (`tests/unit/`)
   - `test_prompt_builder.py`: variable substitution, token truncation (paste 100 messages, verify truncation drops oldest while preserving the most recent user message), `{{char}}` left in place when char_name is None.
   - `test_models.py`: round-trip serialize/deserialize for every domain model with both minimal and maximal field sets.

10. **Smoke test**
    - With Ollama running locally and a model pulled, user can have a 5-turn conversation with Bram and see streaming.
    - Stop Ollama mid-conversation, send a message, verify the UI shows a clear error and recovers when Ollama returns.

# Stage 2 — Multi-Phase Game Loop

**Goal:** Add Phases 1, 2, and 3. The DM narrates with full context, characters respond with restricted context, and the user picks from 4 generated options.

### Tasks

1. **Phase 1 — Director**
   - New module `phases.py` with `run_director(save, scenario, characters)`.
   - Uses the **Director Prompt Assembly** (see top of guide) — NOT the DM prompt assembly. This is a meta-level routing call with a neutral systemic voice.
   - Director system prompt and instruction are hardcoded constants in `phases.py` (not user-editable in the MVP).
   - The character roster is built from active characters using their `description_summary`. If a character has no summary yet, fall back to the first ~150 chars of `description`.
   - If `Scenario.summary` is empty, fall back to the first ~200 chars of `Scenario.initial_message`.
   - **Beat roster**: include only if `scenario.beats` is non-empty AND `save.sandbox_mode` is false. Show the current beat (marked `[CURRENT]`) and all subsequent beats. (For Stage 2 the hardcoded fixture has no beats, so this section is omitted at runtime — but the code path must exist and be exercised by a unit test using a synthetic beats-enabled fixture.)
   - Use Ollama's structured output (`format` parameter with JSON schema):
     ```json
     {
       "type": "object",
       "properties": {
         "dm_should_narrate": {"type": "boolean"},
         "speaker_character_id": {"type": "string"},
         "beat_transition": {"type": "boolean"},
         "next_beat_id": {"type": ["string", "null"]},
         "direction_note": {"type": "string"},
         "reasoning": {"type": "string"}
       },
       "required": ["dm_should_narrate", "speaker_character_id", "beat_transition", "direction_note"]
     }
     ```
   - **Validation**:
     - `speaker_character_id` must be in the active roster; if not, fall back to the first non-DM character.
     - If `beat_transition=true`: `next_beat_id` must be a forward beat ID (current or any later beat in the scenario). If invalid or backward, log a warning, drop the transition, keep the rest of the response.
     - If `beat_transition=false`: ignore `next_beat_id` even if present.
   - The director call should NOT stream — it returns a single small JSON object. Use a non-streaming Ollama call.

2. **Beat transition handler** (between Phase 1 and Phase 2)
   - If the Director's response includes a valid forward `beat_transition`, the orchestrator:
     - Updates `save.current_beat_id = next_beat_id`.
     - Appends a new DM message with `content = beat.starter_prompt` (with `{{user}}`/`{{char}}` substitution applied), `beat_id_at_time = next_beat_id`, and emits an SSE event `event: beat_transition` with `{new_beat_id, new_beat_name}` so the frontend can update the beat header.
     - If `next_beat_id` was the last beat AND the Director also signals the final transition condition met (next call), set `sandbox_mode=true`. (For now: detect this in Stage 3 when ending screens are wired up; in Stage 2 just log it.)
   - If `scenario.beats` is empty, this handler is a no-op.

3. **Phase 2 — Speaker generation**
   - If `dm_should_narrate`: stream DM response first (using DM prompt assembly, including the Director's `direction_note`). Append to messages. Then stream the chosen character's response (character prompt assembly).
   - If not narrating: stream only the chosen character. (The character prompt does NOT receive the Director's note — direction notes are DM-only guidance.)
   - Each streamed segment is its own SSE message group with metadata `{character_id, character_name, beat_id_at_time}`.

4. **Phase 3 — Option drafting**
   - After Phase 2 finishes, run `draft_player_options(save, scenario)`.
   - Uses DM prompt assembly + appended instruction: "Generate exactly 4 plausible next actions or replies the player might take, in {{user}}'s voice. Vary tone (cautious, bold, curious, witty)."
   - Structured output schema:
     ```json
     {"type": "object", "properties": {"options": {"type": "array", "minItems": 4, "maxItems": 4, "items": {"type": "string"}}}, "required": ["options"]}
     ```
   - Send to frontend as a single SSE event: `event: options\ndata: [...]\n\n`.

5. **Frontend updates**
   - Render DM messages with distinct styling (italic, different color block).
   - Render character messages with character name and avatar.
   - After bot turn completes, display 4 option buttons + a "Custom reply..." text input.
   - Clicking an option submits it as the user's next message.
   - Add "regenerate" button on the last AI message group (re-runs Phase 1+2 from the same user turn).
   - Add inline edit on any message (click to edit, save persists; does NOT invalidate later messages).
   - **Beat header (placeholder)**: If `save.current_beat_id` is set, display the beat name in a small header above the chat. In Stage 2 the hardcoded fixture has no beats, so this header stays hidden. Code path must exist.
   - **Resilience-aware SSE handling**: handle `regenerate` (clear buffered tokens, show "Regenerating..."), `validation_warning` (subtle inline warning under the affected message), `validation_failed` (toast), and `error` (toast).

6. **Save model upgrade**
   - Saves now track full `messages[]` per the schema above (including `beat_id_at_time` and `current_beat_id`/`sandbox_mode` fields, even if unused by the Stage 2 fixture).
   - Add `max_context_tokens` field, default 8192, settable via a small UI control in the chat page.

7. **Validation & resilience module** (`validation.py`)
   - Implements the validation + auto-regenerate flow described in the Resilience & Logging section.
   - Validators (one per call type):
     - `validate_director_response(raw, scenario, characters, save) -> Ok|Err`: parses JSON, checks schema, validates `speaker_character_id` is in roster, validates `next_beat_id` if transition is true.
     - `validate_options_response(raw) -> Ok|Err`: parses JSON, ensures `options` is an array of exactly 4 non-empty strings.
     - `validate_streamed_text(buffered, previous_message) -> Ok|Err`: not empty, not whitespace-only, passes loop detection.
   - Retry orchestrator: `with_validation(call_fn, validator, max_retries=2, on_failure=fallback_fn)`.
   - On retry, the call's prompt is augmented with the prior failure reason: `"The previous response failed validation: <reason>. Return ONLY valid JSON matching the schema."`
   - Loop detection helper: `is_loop(new_text, previous_text, same_speaker: bool) -> tuple[bool, float]` returning the decision and similarity ratio.
   - All validation failures, retries, and fallback engagements are logged per the Resilience section.

8. **Tests** (`tests/integration/`, `tests/resilience/`)
   - Build a mock Ollama client (`MockOllamaClient`) with controllable responses: can return valid output, malformed JSON, empty string, repeated identical content, or simulate timeout / mid-stream stall.
   - **Integration tests** (`tests/integration/`):
     - Phase 1 → 2 → 3 happy path with the hardcoded fixture, asserting SSE event sequence.
     - Synthetic two-beat fixture: forced transition, beat header event emitted, `current_beat_id` updates, `starter_prompt` appended.
     - Synthetic four-beat fixture: Director returns `next_beat_id` of beat 3 while current is beat 1 (skipping beat 2), verify it's accepted.
     - Backward-jump rejection: Director returns earlier beat ID, verify rejection logged and rest of response applied.
   - **Resilience tests** (`tests/resilience/`):
     - **Malformed JSON from Director**: mock returns truncated JSON, garbage text, JSON missing required fields. Verify auto-regen triggers, succeeds on retry (mock returns valid the second time), and the system makes forward progress.
     - **All retries fail**: mock returns malformed every time. Verify final fallback engages (DM narrates, 4 generic options, no transition, empty note) and a `validation_failed` SSE event is emitted.
     - **Mid-stream timeout**: mock starts streaming then stalls past the idle timeout. Verify abort, regen, and SSE `regenerate` event.
     - **Empty response**: mock returns empty string for a streamed call. Verify regen.
     - **Loop detection — exact**: mock returns the same content as the previous assistant message. Verify regen.
     - **Loop detection — fuzzy**: mock returns a near-duplicate (90% similar). Verify regen.
     - **Loop detection — exhaustion**: mock returns near-duplicates 3 times in a row. Verify the system gives up after 2 retries and emits the response with a `validation_warning` event.
     - **Ollama unreachable**: mock raises connection error. Verify SSE error event and clean failure (no half-written save messages).

9. **Phase orchestration route**
   - Single endpoint `/api/chat/turn` (SSE) that runs Phase 1 → (beat transition handler) → Phase 2 → Phase 3 in sequence, emitting events as it goes:
     - `event: director` with `{speaker_id, dm_narrating, direction_note}`
     - `event: beat_transition` with `{new_beat_id, new_beat_name}` (only if a transition occurred)
     - `event: token` with `{character_id, text}` (repeated)
     - `event: message_complete` with `{message_id, character_id}`
     - `event: regenerate` with `{reason}` (when validation triggers an auto-retry mid-stream)
     - `event: validation_warning` with `{reason}` (after retries exhausted but content emitted anyway)
     - `event: validation_failed` with `{call, reason}` (after retries exhausted on a structured call; safe fallback used)
     - `event: options` with `{options: [...]}`
     - `event: error` with `{reason}` (Ollama unreachable, etc.)
     - `event: done`

10. **End-to-end resilience smoke test**
    - With a real Ollama instance: run a 10-turn conversation. Confirm INFO logs show every turn, every Director call, and any retries that happened naturally.
    - Stop Ollama mid-stream. Verify clean error in UI and clean log line.

# Stage 3 — Editors, Menus, and Persistence UI

**Goal:** Replace hardcoded fixtures with real CRUD. Build the four interfaces from the spec.

### Tasks

1. **Storage layer**
   - `storage.py`: typed read/write for `data/characters/*.json`, `data/scenarios/*.json`, `data/saves/*.json`.
   - Atomic writes (write to `.tmp`, rename) to prevent corruption.
   - File watching not needed; reload on every read for simplicity.

2. **Character routes**
   - `GET /api/characters` — list.
   - `GET /api/characters/{id}` — fetch one.
   - `POST /api/characters` — create. Triggers background summary generation.
   - `PUT /api/characters/{id}` — update. Triggers background summary regeneration **only if** the new `description` differs from the stored one (compare against `description_hash`).
   - `POST /api/characters/{id}/regenerate-summary` — manual trigger, runs synchronously and returns the new summary.
   - `DELETE /api/characters/{id}` — delete (warn if used by any saved game).
   - `POST /api/characters/import` — accept SillyTavern PNG (read embedded JSON from `chara` tEXt chunk) or `.json` card. Map fields:
     - `name` → `name`
     - `description` + `personality` → `description` (concatenate)
     - `mes_example` → parse into `response_examples` (split by `<START>` blocks, then by `{{user}}:` / `{{char}}:` lines)
     - Default `is_dm` to false.
     - Triggers background summary generation after import.
   - `GET /api/characters/{id}/export` — emit a SillyTavern v2-compatible JSON card. (Summary is a CharacterCadre-specific field; do not export it.)

3. **Summarization module** (new module `summarizer.py`)
   - Used by characters, scenarios, AND beats. Three functions, one shared LLM call pattern:
     - `generate_character_summary(name, description) -> str`
     - `generate_scenario_summary(name, initial_message, system_prompt) -> str`
     - `generate_beat_summary(name, description, transition_condition) -> str`
   - Uses the same Ollama model as gameplay, with focused prompts.
   - Character prompt:
     ```
     Summarize the following character into ONE short sentence (max 20 words) capturing
     their core personality and role. This summary will help a director AI decide when
     this character should speak. Do not repeat the name. Do not add quotes.

     Name: {name}
     Description: {description}
     ```
   - Scenario prompt:
     ```
     Summarize the following scenario into 1-2 short sentences (max 50 words) describing
     the setting, situation, and tone. This summary will orient a director AI that routes
     dialogue between characters. Do not add quotes.

     Name: {name}
     Initial message: {initial_message}
     ```
   - Beat prompt:
     ```
     Summarize the following story beat into ONE short sentence (max 25 words) describing
     what happens in this scene and what the player is doing. This summary will help a
     director AI decide when to advance the plot or skip ahead. Do not add quotes.

     Name: {name}
     Description: {description}
     Transition: {transition_condition}
     ```
   - All three run via FastAPI `BackgroundTasks` so saves return immediately.
   - On failure (Ollama unreachable, parse error), leave the summary empty and log a warning. Phase 1 fallbacks handle the empty case for characters and scenarios; for beats, the Director sees the `name` and `transition_condition` even if `summary` is empty, so it can still function.
   - Hash strategy:
     - Character: SHA-256 of `description` → `description_hash`.
     - Scenario: SHA-256 of `initial_message + system_prompt` → `summary_hash`.
     - Beat: SHA-256 of `name + description + transition_condition` → `summary_hash`.

4. **Scenario routes**
   - Standard CRUD on `/api/scenarios`. Beats are nested inside the scenario document — a single PUT to the scenario updates beats too.
   - `POST /api/scenarios/{id}/regenerate-summary` — manual trigger, runs synchronously.
   - `POST /api/scenarios/{id}/beats/{beat_id}/regenerate-summary` — same for an individual beat.
   - Default new scenarios with a sensible RP system prompt prefilled (provide a constant in code) and an empty `beats` list.
   - On scenario update, for each beat whose `summary_hash` no longer matches its current contents, trigger background regeneration.
   - Background summary generation triggered on create and on update when `initial_message` or `system_prompt` changed.

5. **Save routes**
   - `GET /api/saves` — list (return summary: id, name, scenario name, last updated, message count, current beat name if any).
   - `POST /api/saves` — create new game (body: `scenario_id`, `active_character_ids`, `user_name`). On creation:
     - If scenario has beats: set `current_beat_id = beats[0].id`, append `beats[0].starter_prompt` as the first DM message (with `beat_id_at_time` set).
     - Otherwise: append `Scenario.initial_message` as the first DM message (existing behavior).
   - `GET /api/saves/{id}` — full save.
   - `PUT /api/saves/{id}` — update (name, max_context_tokens, sandbox_mode, etc.).
   - `DELETE /api/saves/{id}`.
   - `POST /api/saves/{id}/advance-beat` — manual beat advance. Body: `{next_beat_id, wipe_context: bool}`.
     - Validates `next_beat_id` is a forward beat in the scenario.
     - If `wipe_context`: replace `messages` with a single DM message containing the new beat's `starter_prompt`. Otherwise: append to existing chat.
     - Updates `current_beat_id` and emits the same state changes as a Director-driven transition.
   - `POST /api/saves/{id}/sandbox-mode` — toggle sandbox mode on/off.

6. **Frontend — Main Menu** (route `/`)
   - Top half: grid of save cards (name, scenario, last played, message count, "Continue" + "Delete" buttons).
   - Bottom half: 3 buttons — "Edit Scenarios", "Edit Characters", "New Game".

7. **Frontend — Edit Characters** (route `/characters`)
   - Right vertical sidebar: list of saved characters with "+ New" button.
   - Main area: form with fields per the model. Response examples are a repeatable pair-input component.
   - "DM" toggle.
   - **Summary field**: editable single-line text input labeled "Director hint (one-sentence summary)". Sits directly under the description textarea. Includes:
     - A small "↻ Regenerate" button that calls `POST /api/characters/{id}/regenerate-summary` and shows a spinner.
     - A subtle status indicator: "Auto-generating..." while a background job is in flight (poll the character endpoint every ~2s after a save until the summary appears or hash matches), "Edited" badge if the user has manually changed it.
     - Helper text: "Used by the Director AI in-game to decide when this persona speaks. Auto-generated on save, but you can edit it directly."
   - "Import from SillyTavern" button (file picker, accepts .png and .json).
   - "Export" button on each saved character.
   - Save / Delete buttons.

8. **Frontend — Edit Scenarios** (route `/scenarios`)
   - Similar layout to Characters.
   - Fields: name, **summary** (textarea — same auto-generation pattern as character summaries: regenerate button, "Auto-generating..." indicator, "Edited" badge, helper text "Used by the Director AI to orient itself to the scenario"), initial message (textarea), system prompt (textarea, pre-filled with default), persistent messages (repeatable list), DM-only info (repeatable list), recommended characters (multi-select chips loaded from character list).
   - **Beats section**: a vertical list of draggable, expandable accordions. Each beat accordion (collapsed view shows order number + name + a small drag handle) expands to show:
     - `name` (text input)
     - `description` (textarea, "Environment, hazards, NPCs in this beat — DM-facing notes")
     - `summary` (single-line input with regenerate button + status indicator + edited badge — same pattern as character/scenario summaries)
     - `transition_condition` (textarea, "Plain English: what player action or story event ends this beat?")
     - `starter_prompt` (textarea, "Prose injected as a DM message when this beat begins. Sets the scene.")
     - "Delete beat" button (red, with confirmation)
   - "+ Add Beat" button at the bottom.
   - Note: if `beats` is empty, the scenario behaves as a single-arc story using `initial_message`. Display a small info box: "No beats defined — this scenario will use 'initial message' as a single opening. Add beats to structure the plot in scenes."
   - When the author drags to reorder, `order` indices renumber automatically. IDs remain stable.

9. **Frontend — New Game** (route `/new-game`)

   - Step 3: User name input.

10. **Frontend — Game** (route `/game/:save_id`)

    - Add header with save name, "back to menu" button.
    - **Beat header**: if `save.current_beat_id` is set, show the active beat name as a subtitle (e.g., "Act II: The Dungeon"). Updates live when an `event: beat_transition` SSE event arrives.

      - "Sandbox mode": toggle, disables Director-driven beat tracking. Calls `POST /api/saves/{id}/sandbox-mode`.
      - "Settings" → `max_context_tokens` slider.
    - **Ending screen**: if `save.sandbox_mode` was just turned on as a result of completing the final beat, show a modal: "The story has reached its conclusion. Continue in sandbox mode, or end the game?" with buttons "Sandbox" and "Return to Main Menu". Manually toggling sandbox mode does not show this — it's only triggered by Director-driven completion.

11. **Quality polish**
    - Loading states everywhere.
    - Empty states with helpful CTAs (e.g., "No characters yet — create one").
    - Confirm dialogs for destructive actions.
    - Keyboard shortcuts: Enter to send, Shift+Enter for newline.

# Cross-Stage Conventions for Claude Code

1. **Always pause for human review at the end of each stage.** Do not proceed to the next stage without explicit approval.
2. **Write tests as you go.** Backend: pytest in three tiers — `tests/unit/` (pure functions: prompt assembly, variable substitution, token truncation, validators), `tests/integration/` (orchestration with mocked Ollama: phases, beat transitions, SSE event sequence), `tests/resilience/` (failure modes: malformed JSON, timeouts, empty responses, loop detection — see Stage 2 task 8). Frontend: at least one integration test per editor (creates → edits → saves) and one for SSE event handling (regenerate, validation_warning).
3. **Logging is part of the build, not an afterthought.** Every new module logs at the levels described in the Resilience & Logging section. Do not ship a stage where logs are missing for the operations introduced in that stage.
4. **Validate every Ollama call.** No structured call goes to the orchestrator without going through `validation.py`. No streaming call completes without the post-stream sanity check. New call types added in future stages require new validators.
5. **Pin dependency versions.** Use lockfiles.
6. **Keep functions small and pure where possible**, especially in `prompt_builder.py`, `phases.py`, and `validation.py` — these are the heaviest-tested modules.
7. **No global state.** Pass `save`, `scenario`, and characters explicitly.
8. **Type everything.** Pydantic models on the backend, TS types on the frontend, ideally generated from the Pydantic schemas (use `pydantic2ts` or hand-mirror them — small enough to maintain).
9. **Ask questions when ambiguous.** Don't guess on edge cases (e.g., what if zero non-DM characters are active? what if the user's max_context_tokens is too low to fit even the system prompt?). Surface these with proposed answers.
10. **Treat all Ollama I/O as untrusted.** Models hallucinate, return malformed JSON, repeat themselves, hang. The Resilience & Logging section is the contract — any new code path that calls Ollama must follow it.
